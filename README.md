# Instantale LLM Proxy — LLMリクエスト置換プロキシMOD

ゲーム(Instantale)がローカルLLMに送る**プロンプト(指示文)を、好きなルールで書き換える**MODです。
世界観・キャラの口調・NPCのふるまいなどを、ゲーム本体をいじらずに調整できます。

例:

> 「あなたはダークファンタジーRPGのキャラクター生成AIだ。」
> → 「あなたは明るく平和なRPGのキャラクター生成AIだ。」

操作するのは `InstantaleLlmProxy.exe`(管理GUI)だけ。ルールの編集・適用/解除・ログ閲覧はすべてこの画面から行えます。

MODのファイルはすべてこの `InstantaleLlmProxy` フォルダ内にまとまっており、ゲームフォルダ直下に余計なものは置きません。

> 内部の仕組みや設定ファイルの詳細など開発者向けの情報は [TECH.md](TECH.md) にまとめています。通常の利用では読む必要はありません。

---

## 導入手順

1. `InstantaleLlmProxy` フォルダごと、ゲームフォルダ (`instantale.exe` がある場所) の直下に置く
2. `InstantaleLlmProxy.exe` を起動して「MOD適用」を押す
3. ゲームを起動する

これだけで動きます。以降はGUIでルールを編集すれば、ゲームを起動したまま次のLLMリクエストから反映されます。

> - 適用時のビルドはWindows標準搭載のC#コンパイラで行うため追加インストール不要
> - 上書き展開すると `llm_replacements.txt` (置換ルール) が初期化されるので、編集済みの場合はバックアップしてから展開すること

## 使い方 (GUI)

`InstantaleLlmProxy.exe` をダブルクリックすると管理GUIが起動します。できることは次のとおりです。

- **対象フォルダの切替** … 「参照...」でゲームフォルダを選択(次回起動時も記憶)。別のゲームフォルダ(本体とコピー等)もこのGUIひとつで管理できる
- **置換ルールの編集/保存** … 表形式で編集。「有効」チェックでルールごとにON/OFF、「置換確率」で確率的な置換、その場で置換テストも可能
  - セル内の改行は編集中に Shift+Enter で入力できる (ファイルへは `\n` の2文字に変換して保存される)
  - **「正規表現」チェック**で置換前を.NET正規表現として解釈 (置換後で `$1` などのキャプチャ参照が使える。詳しくは[正規表現ルール](#正規表現ルール))。正規表現ルールの置換前セルは淡黄色で表示される
  - **「メモ」列**にルールごとの覚え書きを残せる (置換動作には影響しない)
- **タブでの整理** … ルールをタブで分類 (「タブ追加/タブ名変更/タブ削除」)。タブ内のチェックでタブごと一括ON/OFF。タブ見出しはドラッグで並び替え可能 (順序は保存時に反映)
  → 「キャラ生成」「実験中」など目的別のルールセットを丸ごと切り替えられる
- **すべてタブ** … 先頭の「すべて」タブで全タブのルールを横断表示・検索 (タブ名/置換前/置換後)。行をダブルクリックすると該当タブの該当行へジャンプ (「すべて」タブ自体は閲覧専用)
- **適用/解除・プロセス管理** … MODの適用/解除、llama-serverプロセスの状態表示と強制終了
- **プロンプト圧縮の切替** … 「プロンプト圧縮」チェックで、LLMへ送るプロンプトの自動軽量化をON/OFF(既定ON・基本はそのままでOK)。応答が返らない/途中で切れるといった不調の切り分け時に一時的にOFFにできる (詳しくは[プロンプト内スキーマ説明のコンパクト化](TECH.md#プロンプト内スキーマ説明のコンパクト化))
- **ログ閲覧** … `llm_proxy.log` を自動更新で表示
- **OpenAI互換メニュー** … ゲーム同梱のローカルLLMの代わりに、外部のOpenAI互換サーバを使うための設定 (既定はOFF・使わない場合は無視してよい。詳しくは[任意のOpenAI互換サーバを使う](#任意のopenai互換サーバを使う-任意))
- **設定メニュー** … 上部の「設定」→「機能設定...」で、自動的な安定化・軽量化機能(JSONFIX/DEDUP/EVENTLOG/シングルトン化)を**1つずつON/OFF**できる (既定はすべてON・即時反映・ゲーム再起動不要)。「デバッグ設定...」では、ログに出す項目を1つずつON/OFFできる
  - **動作ログ** (`llm_proxy.log`・既定はすべてON) … 置換 `[REPLACE]`/`[SKIP]`、プロンプト圧縮 `[COMPACT]`、重複ブロックの畳み込み `[DEDUP]`、イベント履歴ログの圧縮 `[EVENTLOG]`、JSON安定化 `[JSONFIX]`、ルールファイルの読込 `[RULES]`、OpenAI互換サーバへの中継 `[OPENAI]`
  - **調査用の診断ログ・ダンプ** (既定はすべてOFF) … 診断行 `[DIAG]`、`llm_proxy_schema_dump.log` / `llm_proxy_prompt_dump.log` / `llm_proxy_resp_dump.log` の各ダンプ。ログが非常に大きくなるため調査時のみON (詳しくは[調査用の診断ログ](TECH.md#調査用の診断ログ-デフォルトoff))
  - 起動・エラー等の重要ログ (`[BOOT]`/`[ERROR]` 等) はどの項目にも属さず**常に記録**される

> - ラッパー自体はゲームが起動時に自動で立ち上げ、終了時に終わらせるのでGUIから起動する必要はない。
> - 対象が Program Files 配下の場合、適用/解除やルール保存には管理者権限が必要なことがある (その場合はGUIを管理者として実行)。ラッパーがログを書けない場合は `%LOCALAPPDATA%\llm_proxy.log` に出力される (GUIは自動表示)。

## 置換ルールの手動編集

GUIを使わずに `llm_replacements.txt` を直接編集することも可能。

```
置換前=>置換後=>置換確率
```

- 1行1ルール、UTF-8、行頭 `#` はコメント
- 改行を含めたい場合は `\n` (バックスラッシュ+n の2文字) と書く。LLMリクエストのJSON内の改行エスケープにそのまま一致する。GUIのグリッド上では実際の改行として表示される
- 置換確率は 0〜100 の整数。省略すると 100 (常に置換)
- 同じ「置換前」のルールが複数ある場合、確率に応じてどれか1つの「置換後」が選ばれる (タブをまたいでも同様)
  - 確率の合計が100以下のとき: 残りの確率では置換しない (例: 30と30なら4割は無置換)
  - 確率の合計が100を超えるとき: 各ルールの `値÷合計` の割合で必ずどれかに置換される
- 行頭に `#off:` を付けるとそのルールだけ無効化 (GUIの「有効」チェックと連動)
- 行頭に `regex:` を付けると正規表現ルール (GUIの「正規表現」チェックと連動。無効化と併用する場合は `#off:regex:...` の順)
- `#memo:メモ` の行は直後のルールの覚え書き (GUIのメモ欄と連動。置換には影響しない)
- `#tab:タブ名` の行から次のタブ行までが1つのタブ (GUIのタブと連動)。`#offtab:タブ名` はタブごと無効化で、中のルールはすべて無視される。タブ行が無いファイルは「標準」タブとして読み込まれる
- ゲーム起動中に編集・保存してもOK。次のLLMリクエストから自動反映される
- 置換された内容は `llm_proxy.log` に記録される

### 正規表現ルール

GUIの「正規表現」チェック (ファイル上は行頭 `regex:`) を付けると、置換前を**.NET正規表現のパターン**として解釈します。表記ゆれ (数値・名前の違いなど) を1ルールでまとめて置換したいときに便利です。

```
regex:口調: (丁寧|敬語)=>口調: くだけた話し方
regex:HPが(\d+)残っている=>HPが$1ポイント残っている
```

- 置換後では `$1` `$2` でキャプチャした部分を参照できる (`$` 自体を出力したいときは `$$`)
- プロンプト中の改行 (JSON内の `\n` の2文字) にマッチさせるには、パターンに `\\n` (バックスラッシュ2つ+n) と書く。正規表現ルールでは置換前のセル内改行 (Shift+Enter) は使えない
- 通常ルールにある `\uXXXX` エスケープ形式への自動対応は、正規表現ルールには**無い** (機械変換するとパターンの意味が変わるため)。エスケープ形式のプロンプトに当てたい場合はパターン側に `\\uXXXX` を自分で書く
- 不正なパターンは保存時にGUIがチェックして知らせる (直接編集した場合は読込時に `[WARN]` ログで無視される)。照合には1秒のタイムアウトがあり、暴走パターンで中継が止まることはない
- 確率・タブ・有効/無効の扱いは通常ルールと同じ。同じ置換前文字列でも正規表現とリテラルは別ルールとして扱われる

## 任意のOpenAI互換サーバを使う (任意)

ゲーム同梱のローカルLLMの代わりに、**任意のOpenAI互換サーバ**を使えます。OpenAI本家に限らず、LM Studio / Ollama / vLLM / 各種ゲートウェイなど `/v1/chat/completions` を話すものなら指定できます。**設定しなければ従来どおりローカルLLMで動くので、使わない人は何もしなくて構いません (既定OFF)。**

接続先はGUIのメニュー「OpenAI互換」→「接続設定...」で入力します (エンドポイント / モデル名 / APIキー(任意))。**接続設定は保存しただけでは動作が変わりません** — 下の(1)か(2)を選んで初めて使われるので、設定を残したままローカルLLMへ戻せます。

置換ルール (`llm_replacements.txt`) は(1)(2)どちらの経路でも適用されます。

### (1) ラッパー翻訳 … ゲームはローカルLLM設定のまま

MOD適用済みのラッパーが、ゲームの出すローカルLLM向けリクエストをOpenAI形式へ**翻訳して**接続先へ送ります。ゲーム側の設定は触りません。

1. 「接続設定...」で エンドポイント と モデル名 を入力 (この経路では両方必須)
2. メニューの「ラッパー翻訳を使用」にチェック (または接続設定ダイアログ先頭の動作モードで選択)
3. ゲーム (llama-server) を起動し直す

有効な間はゲーム同梱のローカルLLM (`llama-server-real.exe`) を起動しません。チェックを外せばローカルLLMに戻ります。

### (2) 中継サーバ … ゲーム側のOpenAI互換設定を使う

ゲームにもOpenAI互換の設定欄がある場合はこちら。プロキシを待ち受けサーバとして動かし、その先へ中継します。ゲームはOpenAI形式で話すため翻訳は行わず、置換ルールだけ適用して素通しします。

1. 「接続設定...」で エンドポイント と 待ち受けポート (既定 `8181`) を設定
2. メニューの「中継サーバを起動」
3. ゲーム内のOpenAI互換設定のエンドポイントに `http://127.0.0.1:<待ち受けポート>/v1` を指定

モデル名とAPIキーはゲーム側の指定が優先され、空のときだけ接続設定の値で補われます。中継サーバはGUIを閉じても動き続けます (停止はメニューから)。この経路ではゲームがllama-serverを起動しないため、MOD適用の有無は無関係です。

> - 現在どのLLMで動く設定になっているかは、GUIの「状態と操作」欄の**「LLM動作」行**に常時表示されます
> - 設定変更の反映には、ラッパー翻訳は**ゲームの起動し直し**、中継サーバは**中継サーバの再起動**が必要です (置換ルールと違い自動再読込はしません)
> - APIキーは `llm_proxy_settings.ini` に平文で保存されます。共有前にご注意ください
> - 外部サーバが落ちてもローカルLLMへ自動では戻りません (勝手にモードが変わらないようにするため)。GUIで切り替えてください

## 自動的な安定化・軽量化 (既定ON)

置換ルールとは別に、ゲーム側の挙動に起因する不具合を避けるための処理をバックグラウンドで自動的に行っています。いずれも既定ONで、無効化はGUIの「設定」→「機能設定...」から**1つずつチェックを外すだけです** (即時反映・ゲーム再起動不要)。詳しい仕組みは[TECH.md](TECH.md)を参照。

- **プロンプト圧縮** … スキーマ説明の定型句を圧縮 (GUIの「プロンプト圧縮」チェックでON/OFF)。あわせて次の2つも同様にプロンプトの肥大化を防ぎます
  - **クエストイベント履歴ログの圧縮** … イベント判定用プロンプトに溜まり続ける古いターンの履歴を直近3ターンだけに削減 (詳しくは[TECH.md](TECH.md#クエストイベント履歴ログの圧縮-eventlog))
  - **プロンプト重複ブロックの畳み込み** … 再生成のたびに重複していくスキーマ指示ブロックを1個に畳んでコンテキスト枯渇を防止 (詳しくは[TECH.md](TECH.md#プロンプト重複ブロックの畳み込み-dedup))
- **本物 (llama-server) の多重起動抑止** … 同時に何本も起動されがちな本体プロセスを1本に集約し、VRAM枯渇やロード遅延を防止 (詳しくは[TECH.md](TECH.md#本物-llama-server-の多重起動抑止))
- **JSON安定化 (JSONFIX)** … 万一ゲーム側がjson_schemaを送っていないリクエストを検出した場合に備えた安全網 (詳しくは[TECH.md](TECH.md#json安定化-jsonfix))

## 注意

- ゲームのアップデートで `bin\` 内の `llama-server.exe` が上書きされたら「MOD適用」を再実行してください
- 置換はLLMへの「入力」のみ。LLMからの応答文は変更しません
- ゲームがクラウド/外部のLLMをそのまま使う設定 (プロキシを経由しない) の場合は効きません。外部のOpenAI互換サーバでも置換を効かせたい場合は[上記の中継](#任意のopenai互換サーバを使う-任意)を使ってください

## ファイル構成

```
InstantaleLlmProxy\
├─ InstantaleLlmProxy.exe   … 管理GUI (操作するのはこれだけ)
├─ llm_replacements.txt     … 置換ルール (保存すると即時反映・再起動不要)
├─ llm_proxy_settings.ini   … 動作設定 (プロンプト圧縮・安定化機能のON/OFF・ログ項目・OpenAI互換の接続先。GUI操作で更新)
├─ llm_proxy.log            … 動作ログ ([REPLACE] 行が置換の記録・実行時に生成)
├─ README.md                … このファイル (利用者向け)
├─ TECH.md                  … 技術仕様 (仕組み・設定ファイル詳細など開発者向け)
├─ LICENSE                  … ライセンス (MIT)
└─ src\                     … ソースと開発者向けスクリプト (通常は触らなくてよい)
   ├─ proxy\                … ラッパーのソース (適用時にGUIがビルドする。機能ごとにファイル分割、全体で1つの partial class LlmProxy)
   ├─ gui\                  … 管理GUIのソース (機能ごとにファイル分割、全体で1つの partial class MainForm)
   ├─ llm_proxy_apply.bat   … MOD適用 (コマンドライン派・GUIと同等。proxy\*.cs をまとめてビルド)
   ├─ llm_proxy_revert.bat  … MOD解除 (コマンドライン派・GUIと同等)
   └─ llm_proxy_gui.bat     … GUIをソースから再ビルドして起動 (開発用。gui\*.cs をまとめてビルド)
```
