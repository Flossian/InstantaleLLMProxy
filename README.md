# InstantaleLlmProxy — LLMリクエスト置換プロキシMOD

ゲーム(Instantale)がローカルLLMに送る**プロンプト(指示文)を、好きなルールで書き換える**MODです。
世界観・キャラの口調・NPCのふるまいなどを、ゲーム本体をいじらずに調整できます。

例:

> 「あなたはダークファンタジーRPGのキャラクター生成AIだ。」
> → 「あなたは明るく平和なRPGのキャラクター生成AIだ。」

操作するのは `InstantaleLlmProxy.exe`(管理GUI)だけ。ルールの編集・適用/解除・ログ閲覧はすべてこの画面から行えます。

MODのファイルはすべてこの `InstantaleLlmProxy` フォルダ内にまとまっており、ゲームフォルダ直下に余計なものは置きません。

---

## 導入手順

1. `InstantaleLlmProxy` フォルダごと、ゲームフォルダ (`instantale.exe` がある場所) の直下に置く
2. `InstantaleLlmProxy.exe` を起動して「MOD適用」を押す
3. ゲームを起動する

これだけで動きます。以降はGUIでルールを編集すれば、ゲームを起動したまま次のLLMリクエストから反映されます。

> - 適用時のビルドはWindows標準搭載のC#コンパイラで行うため追加インストール不要
> - 上書き展開すると `llm_replacements.txt` (置換ルール) が初期化されるので、編集済みの場合はバックアップしてから展開すること

## 使い方 (GUI)

`InstantaleLlmProxy.exe` をダブルクリックすると管理GUIが起動します。できることは次のとおりです。

- **対象フォルダの切替** … 「参照...」でゲームフォルダを選択(次回起動時も記憶)。別のゲームフォルダ(本体とコピー等)もこのGUIひとつで管理できる
- **置換ルールの編集/保存** … 表形式で編集。「有効」チェックでルールごとにON/OFF、「置換確率」で確率的な置換、その場で置換テストも可能
  - セル内の改行は編集中に Shift+Enter で入力できる (ファイルへは `\n` の2文字に変換して保存される)
- **タブでの整理** … ルールをタブで分類 (「タブ追加/タブ名変更/タブ削除」)。タブ内のチェックでタブごと一括ON/OFF
  → 「キャラ生成」「実験中」など目的別のルールセットを丸ごと切り替えられる
- **すべてタブ** … 先頭の「すべて」タブで全タブのルールを横断表示・検索 (タブ名/置換前/置換後)。行をダブルクリックすると該当タブの該当行へジャンプ (「すべて」タブ自体は閲覧専用)
- **適用/解除・プロセス管理** … MODの適用/解除、llama-serverプロセスの状態表示と強制終了
- **ログ閲覧** … `llm_proxy.log` を自動更新で表示

> - ラッパー自体はゲームが起動時に自動で立ち上げ、終了時に終わらせるのでGUIから起動する必要はない。
> - 対象が Program Files 配下の場合、適用/解除やルール保存には管理者権限が必要なことがある (その場合はGUIを管理者として実行)。ラッパーがログを書けない場合は `%LOCALAPPDATA%\llm_proxy.log` に出力される (GUIは自動表示)。

## 置換ルールの手動編集

GUIを使わずに `llm_replacements.txt` を直接編集してもかまいません。

```
置換前=>置換後=>置換確率
```

- 1行1ルール、UTF-8、行頭 `#` はコメント
- 改行を含めたい場合は `\n` (バックスラッシュ+n の2文字) と書く。LLMリクエストのJSON内の改行エスケープにそのまま一致する。GUIのグリッド上では実際の改行として表示される
- 置換確率は 0〜100 の整数。省略すると 100 (常に置換)
- 同じ「置換前」のルールが複数ある場合、確率に応じてどれか1つの「置換後」が選ばれる (タブをまたいでも同様)
  - 確率の合計が100以下のとき: 残りの確率では置換しない (例: 30と30なら4割は無置換)
  - 確率の合計が100を超えるとき: 各ルールの `値÷合計` の割合で必ずどれかに置換される
- 行頭に `#off:` を付けるとそのルールだけ無効化 (GUIの「有効」チェックと連動)
- `#tab:タブ名` の行から次のタブ行までが1つのタブ (GUIのタブと連動)。`#offtab:タブ名` はタブごと無効化で、中のルールはすべて無視される。タブ行が無いファイルは「標準」タブとして読み込まれる
- ゲーム起動中に編集・保存してもOK。次のLLMリクエストから自動反映される
- 置換された内容は `llm_proxy.log` に記録される

## 注意

- ゲームのアップデートで `bin\` 内の `llama-server.exe` が上書きされたら「MOD適用」を再実行してください
- 置換はLLMへの「入力」のみ。LLMからの応答文は変更しません
- クラウドLLM(OpenAI API等)使用時は経由しないので効きません

## ファイル構成

```
InstantaleLlmProxy\
├─ InstantaleLlmProxy.exe   … 管理GUI (操作するのはこれだけ)
├─ llm_replacements.txt     … 置換ルール (保存すると即時反映・再起動不要)
├─ llm_proxy.log            … 動作ログ ([REPLACE] 行が置換の記録・実行時に生成)
├─ README.md                … このファイル
└─ src\                     … ソースと開発者向けスクリプト (通常は触らなくてよい)
   ├─ llm_proxy.cs          … ラッパーのソース (適用時にGUIがビルドする)
   ├─ llm_proxy_gui.cs      … 管理GUIのソース
   ├─ llm_proxy_apply.bat   … MOD適用 (コマンドライン派・GUIと同等)
   ├─ llm_proxy_revert.bat  … MOD解除 (コマンドライン派・GUIと同等)
   └─ llm_proxy_gui.bat     … GUIをソースから再ビルドして起動 (開発用)
```

# 技術仕様

> 以下は仕組みを詳しく知りたい人向けです。**通常の利用では読む必要はありません。**

## 仕組み

ゲームは `bin\llama-b7054-bin-win-*\llama-server.exe` を毎回空きポートで起動し、HTTP (`/apply-template`, `/completion`) で通信しています。
このMODは `llama-server.exe` を「置換プロキシ内蔵ラッパー」に差し替えます。

```
ゲーム → [llama-server.exe (ラッパー/プロキシ)] → [llama-server-real.exe (本物)]
```

- ラッパーは本物を別の空きポートで起動し、ゲーム指定のポートで待ち受ける
- リクエストボディ(JSON)内の文字列をルールに従って置換して中継
- レスポンス(ストリーミング含む)は無加工で素通し
- 日本語が `\uXXXX` エスケープされて送られる場合にも自動対応
- `llm_replacements.txt` の変更はリクエストのたびに自動検知して再読込
  (ゲームを起動したまま保存すれば次のLLMリクエストから反映される)
- ラッパーが終了/killされると本物も道連れで終了する (プロセス残留なし)
- 適用時に `bin\llama-*\` へ `llm_proxy_dir.txt` (このMODフォルダの場所) を書き、ラッパーはそれを見てルールファイルの場所を確実に特定する。そのためMODフォルダの名前は自由に変えてよい (解除時に自動削除される)

## JSON安定化 (JSONFIX)

**JSON安定化 (JSONFIX)** は、LLMの構造化出力(JSON)が壊れる問題に備える機能です。

ゲームはプロンプト内にPython dict形式のスキーマを埋め込み、自然言語で「JSON形式で出力せよ」とLLMに指示しています。ただし**実地調査の結果、ゲーム自身も `/completion` リクエストで llama-server の `json_schema` パラメータを送っており、文法制約付き生成 (grammar-constrained decoding) を有効化していることが分かりました**。そのため構文的なJSON破損(閉じ忘れ・型違反・スキーマ外キー)は、少なくとも検証した範囲では起こりにくい状態です。

このMODのJSONFIXは、万一ゲーム側が `json_schema` を送っていないリクエストを検出した場合に備えた**安全網**です:

- `/completion` リクエストのプロンプト内に埋め込まれた Python dict形式のスキーマを検出して本物のJSON Schemaに変換し、`json_schema` パラメータとして注入します
- ゲーム側が既に `json_schema` / `grammar` を指定している場合は**何もしません**(現状のほとんどのリクエストがこれに該当し、正常です)
- スキーマの解析に失敗した場合は、汎用JSON文法(GBNF)で最低限の構文だけを保証します
- スキーマ指示の無いリクエスト (ナレーション等の自由文出力) には作用しません
- 無効化したい場合: MODフォルダに `llm_proxy_jsonfix_off.txt` という名前の空ファイルを置く (削除すれば再有効化。ゲーム再起動不要)

### JSON安定化・プロンプト最適化ルール (同梱)

`llm_replacements.txt` には、以下のタブをあらかじめ同梱しています (GUIでON/OFF可能)。いずれも実際にログで確認できた不具合への対処なので、有効のままにしておくことを推奨します。

- **JSON安定化** タブ: プロンプト内の `True`/`False`/`None` といったPython表記の指示を `true`/`false`/`null` に正し、「スキーマに無いキーの出力禁止・コードブロック禁止」を明示する置換
- **プロンプト最適化** タブ: ゲーム内プロンプトの重複した指示文・壊れた番号付け・誤字(`- 口調: None` の生漏れ等)の修正、冗長表現の簡潔化

### 調査用の診断ログ (デフォルトOFF)

実際にJSON破損らしき不具合(応答が返らない・NPCの発言がおかしい・処理が止まる等)を見た場合は、MODフォルダに `llm_proxy_diag_on.txt` という空ファイルを置くと、以降のリクエストについて

- `llm_proxy.log` に `[DIAG]` 行(受信キー一覧・`n_predict`・プロンプト文字数)
- `llm_proxy_schema_dump.log` にゲームが実際に送った `json_schema` の中身全体

が記録されるようになります(ゲーム再起動不要)。不具合が起きた前後の時刻のログを見比べることで、打ち切り(`n_predict`不足)なのかスキーマの複雑さに起因するのかを切り分けられます。調査が終わったらファイルを削除すれば元のログ量に戻ります。
